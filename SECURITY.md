| Version | Supported |
|---------|-----------|

## Reporting a Vulnerability

If you discover a security vulnerability in Relic itself (not findings from using Relic against authorized targets), please report it responsibly.

### How to Report

1. **Do NOT open a public GitHub issue** for security vulnerabilities
2. Email: Open a private security advisory via GitHub's [Security Advisories](https://github.com/NagusameCS/Relic/security/advisories/new)
3. Include:
   - Description of the vulnerability
   - Steps to reproduce
   - Potential impact
   - Suggested fix (if any)

### What Qualifies

- Vulnerabilities in Relic's code (scope bypass, injection in API, etc.)
- Insecure defaults that could lead to unintended behavior
- Dependencies with known CVEs that affect Relic's functionality

### What Does NOT Qualify

- Findings from using Relic's modules against authorized targets (that's the tool working as intended)
- Missing "evasion" features — Relic intentionally does not include detection evasion
- Issues with Ollama, Vagrant, VirtualBox, or other third-party dependencies (report those upstream)

### Response Timeline

- **Acknowledgment:** Within 48 hours
- **Assessment:** Within 7 days
- **Fix (if confirmed):** Within 30 days for critical, 90 days for others

## Security Architecture

### Scope Enforcement

Relic enforces an allowlist-based scope system at the engine level. Every command — whether generated by the LLM or entered by the user — is checked against `authorized_targets` before execution. Commands referencing hosts, IPs, or URLs outside the scope are blocked. This is defense-in-depth: the LLM system prompt also instructs the model to stay in scope, but the engine check is the hard enforcement layer.

### VM Isolation

All offensive commands execute inside isolated virtual machines (Vagrant/VirtualBox). The host system is never directly exposed to pentesting tools or their side effects.

### Local-Only Processing

- LLM inference runs locally via Ollama (`127.0.0.1:11434`)
- Web API listens on localhost only (`127.0.0.1:8746`)
- Session data stored on local disk (`~/.relic/sessions/`)
- No telemetry, analytics, or cloud API calls

### Known Limitations

1. **Scope enforcement is string-matching based.** Sophisticated evasion (e.g., encoding an IP as hex in a curl command) could theoretically bypass it. The regex-based fallback catches most patterns but is not exhaustive.

2. **The LLM can hallucinate.** It may generate commands that don't exist, have incorrect syntax, or target the wrong host. Scope enforcement catches unauthorized targets, but nonsensical commands will still execute (and fail) in the VM.

3. **The Web API has no authentication.** It binds to `127.0.0.1` only, so it's not network-accessible by default. If you expose it on `0.0.0.0`, anyone on your network could control it. Don't do this.

4. **Session files are unencrypted JSON.** They may contain sensitive data (credentials, vulnerabilities, command output). Treat `~/.relic/sessions/` as sensitive.

5. **CORS is set to `*`.** This is required for the GitHub Pages static UI to connect to the local API. It does not pose a risk because the API only listens on localhost.
